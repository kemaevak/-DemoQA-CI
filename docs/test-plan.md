# Тест-план: сайт DemoQA

## 1. Введение

Документ описывает план тестирования учебного веб-приложения
[demoqa.com](https://demoqa.com). Сайт представляет собой набор
интерактивных UI-компонентов: формы, таблицы, чекбоксы, алерты,
фреймы, виджеты и элементы drag-and-drop.

Цель проекта — провести функциональное тестирование основных разделов
сайта, задокументировать найденные дефекты и автоматизировать
регрессионные проверки.

## 2. Объект тестирования

| Параметр | Значение |
|---|---|
| Приложение | DemoQA |
| URL | https://demoqa.com |
| Тип | Веб-приложение (SPA на React) |
| Требования | Формальных требований нет, ожидаемое поведение определяется здравым смыслом и общепринятыми UX-практиками |

## 3. Область тестирования

### В объёме (in scope)

- **Elements** — Text Box, Check Box, Radio Button, Web Tables, Buttons,
  Links, Upload and Download, Dynamic Properties;
- **Forms** — Practice Form;
- **Alerts, Frame & Windows** — Browser Windows, Alerts, Frames, Modal Dialogs;
- **Widgets** — Accordion, Select Menu, Progress Bar, Slider, Date Picker, Tool Tips;
- **Interactions** — Droppable.

### Вне объёма (out of scope)

- Раздел **Book Store Application** (API-тестирование планируется как
  отдельный проект);
- нагрузочное тестирование;
- тестирование безопасности;
- кроссбраузерное тестирование (проверки выполняются в Chromium);
- тестирование вёрстки на мобильных разрешениях.

## 4. Подход к тестированию

1. **Исследовательское тестирование** — знакомство с разделами сайта,
   составление чек-листов.
2. **Ручное функциональное тестирование** — прохождение тест-кейсов,
   оформление баг-репортов на найденные дефекты.
3. **Автоматизация** — покрытие регрессионных сценариев автотестами
   (Python + pytest + Playwright, паттерн Page Object).
4. **CI** — автоматический запуск тестов в GitHub Actions при каждом
   изменении кода.

Используемые техники тест-дизайна: эквивалентное разбиение, анализ
граничных значений (длина телефонного номера, возраст в таблице),
негативное тестирование форм.

## 5. Критерии

### Критерии начала тестирования

- сайт доступен;
- составлены чек-листы по разделам из области тестирования.

### Критерии окончания тестирования

- пройдены все тест-кейсы;
- все найденные дефекты задокументированы;
- регрессионные сценарии покрыты автотестами, тесты проходят стабильно.

## 7. Риски

| Риск | Влияние | Как учтён |
|---|---|---|
| Сайт учебный, отвечает медленно и нестабильно | Ложные падения автотестов | Увеличены таймауты, настроен перезапуск упавших тестов |
| Рекламные баннеры перекрывают элементы | Ложные падения, невозможность кликнуть | В автотестах блокируются рекламные домены |
| Часть дефектов заложена на сайте намеренно (сайт создан для обучения тестировщиков) | Нельзя отличить намеренный баг от случайного | Все отклонения фиксируются как дефекты — с точки зрения пользователя разницы нет |
